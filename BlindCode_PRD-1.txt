1. EXECUTIVE SUMMARY
================================================================================

BlindCode is a VS Code extension that transforms the coding experience for
visually impaired developers. Unlike screen readers bolted onto sighted tools,
BlindCode is architected from the ground up assuming zero vision. Code has
spatial identity — depth, position, and texture communicated through audio.
An AI co-pilot, wired directly into the VS Code runtime, navigates, diagnoses,
and edits code through natural conversation. Every AI action passes through a
4-layer spoken trust protocol — the developer retains full control at all times.

Built entirely with free, open-source tools. $0 infrastructure cost.

Target Event   : Hackathon (3000+ teams)
Team Skill     : Software / AI Development
Platform       : VS Code Extension
Total Cost     : $0
Demo Style     : Blindfolded live coding
Demo Duration  : 5-10 minutes
Expected Result: First Prize

2. PROBLEM STATEMENT
================================================================================

2.1 The Scale
-------------
- 253 million people globally are visually impaired (WHO)
- 2.2 billion have some form of vision impairment
- An estimated zero purpose-built IDEs exist for blind developers

2.2 Current State is Broken
----------------------------
Every existing developer tool (VS Code, IntelliJ, Sublime, etc.) was designed
assuming the user can see. Accessibility features are retrofitted patches:

  - Screen readers (NVDA, JAWS, VoiceOver) describe what sighted developers
    see — raw HTML aria labels, tab indexes, visual element names
  - Navigating to a function in VS Code + NVDA requires 3+ minutes of
    tab-pressing through raw accessibility tree
  - GitHub Copilot and Claude Voice are general AI tools with no awareness
    of cursor position, file structure, live errors, or project context
  - No tool gives code spatial identity — blind developers read code
    linearly, line by line, with no structural awareness

2.3 The Result
--------------
Most visually impaired people never attempt software development. Not because
of lack of intelligence or ambition — because the tooling excludes them
entirely before they write a single line.

2.4 The Opportunity
-------------------
BlindCode is not an accessibility patch. It is a new paradigm of how humans
interact with code — one that happens to be designed for blind developers first,
and will benefit all developers who want eyes-free coding.

5. PRODUCT OVERVIEW
================================================================================

BlindCode is a VS Code Extension consisting of 5 integrated subsystems:

  [1] Spatial Audio Engine
      Code as a 3D soundscape — depth, position, texture through audio

  [2] Conversational AI Co-Pilot
      Natural language development — speak intent, not commands

  [3] Trust Protocol Layer
      4-stage safety system — AI never acts without developer approval

  [4] Code GPS Navigation
      Spoken landmarks, breadcrumbs, directional movement through code

  [5] Audio Debugger
      Plain English error narration, variable states, execution replay

All 5 subsystems work simultaneously and integrate with VS Code's Language
Server Protocol for live access to errors, diagnostics, and file state.

10. SPATIAL AUDIO ENGINE SPECIFICATION
================================================================================

10.1 Pitch Mapping (Nesting Depth)
------------------------------------
  Depth 0 (global scope)   →  100 Hz  (deep, grounding tone)
  Depth 1 (function)       →  200 Hz  (mid tone, comfortable)
  Depth 2 (if/loop)        →  300 Hz  (higher, alert)
  Depth 3 (nested block)   →  400 Hz  (high, warning signal)
  Depth 4+ (deep nesting)  →  500 Hz+ (critical, complexity warning)

10.2 Stereo Positioning (File Location)
-----------------------------------------
  Line 1 (top of file)     →  Hard left  (-1.0 stereo)
  Line 50% (middle)        →  Center     ( 0.0 stereo)
  Last line (bottom)       →  Hard right (+1.0 stereo)
  Formula: stereo = (currentLine / totalLines) * 2 - 1

10.3 Audio Texture (Syntax Type)
----------------------------------
  Normal code              →  Clean sine tone
  Error                    →  Buzzing repeating pulse (3Hz)
  Warning                  →  Soft intermittent buzz (1Hz)
  Comment                  →  Soft chime (high, brief)
  Function boundary        →  Sharp click (entry/exit)
  Class definition         →  Warm sustained pad
  Import statement         →  High bright ping
  End of file              →  Descending tone sequence

10.4 Movement Sounds
---------------------
  Moving down (next line)  →  Subtle tick sound
  Moving up (prev line)    →  Reverse tick sound
  Jumping to function      →  Whoosh + arrival chime
  Entering a block         →  Pitch rises smoothly
  Exiting a block          →  Pitch drops smoothly

10.5 Implementation (Howler.js)
---------------------------------

  // Core spatial audio call
  const sound = new Howl({
    src: ['base-tone.mp3'],
    stereo: calculateStereoPosition(currentLine, totalLines),
    rate: calculatePitchRate(nestingDepth),    // rate multiplies base pitch
    volume: 0.6
  });
  sound.play();

  // Pitch rate table (rate * base frequency = perceived pitch)
  Depth 0  →  rate: 1.0
  Depth 1  →  rate: 1.5
  Depth 2  →  rate: 2.0
  Depth 3  →  rate: 2.5
  Depth 4+ →  rate: 3.0

  // Error pulse — plays on loop while error line is focused
  const errorPulse = new Howl({
    src: ['error-buzz.mp3'],
    loop: true,
    volume: 0.4
  });

10.6 Audio Files Required
---------------------------
  base-tone.mp3        —  500ms sine wave, 200Hz
  error-buzz.mp3       —  200ms buzz, repeating
  chime.mp3            —  100ms high chime
  click.mp3            —  50ms sharp click
  whoosh.mp3           —  300ms navigation whoosh
  arrival.mp3          —  150ms arrival chime
  warning-buzz.mp3     —  300ms soft intermittent buzz
  eof-tone.mp3         —  600ms descending sequence

  All audio files: generate free with Audacity or download from freesound.org

11. AI BRAIN — 5 LAYER ARCHITECTURE
================================================================================

11.1 Layer 1 — Project Context Injection
------------------------------------------
Every API call to Claude/Gemini automatically includes:

  System Prompt Template:
  -----------------------
  You are BlindCode, an AI coding co-pilot for a blind developer.
  The developer cannot see the screen. You must:
  - Always describe code in plain spoken English, not visual terms
  - Never say "as you can see" or "look at line X"
  - Always narrate spatial relationships (inside, outside, above, below)
  - State confidence level (0-100%) before every suggestion
  - Keep responses concise — developer hears every word in real time
  - Propose changes, never apply them — wait for "confirm"

  Current Context (injected per request):
  ----------------------------------------
  File: {filename} ({total_lines} lines)
  Position: Line {current_line}, Column {current_column}
  Scope: Inside {current_function} → inside {current_class}
  Errors: {error_list}
  Warnings: {warning_list}
  Last 5 Actions: {action_history}
  Available Checkpoints: {checkpoint_names}
  Surrounding Code (lines {n-5} to {n+5}):
  {code_snippet}

11.2 Layer 2 — Audio Intent Parser
-------------------------------------
Converts natural language to structured commands:

  Input: Raw speech transcript
  Output: JSON intent object

  Intent Types and Examples:
  ---------------------------
  NAVIGATE
    "Take me to the login function"
    "Go to where I last edited"
    "Jump to line 42"
    → { type: "NAVIGATE", target: "function", name: "login" }

  DIAGNOSE
    "Does anything look wrong?"
    "Something feels broken here"
    "What's wrong with this function?"
    → { type: "DIAGNOSE", scope: "current_function" }

  EDIT
    "Fix it"
    "Add a null check before user.id"
    "Make this function async"
    → { type: "EDIT", description: "add null check", target: "user.id" }

  QUERY
    "What does this function do?"
    "Where am I?"
    "How complex is this code?"
    → { type: "QUERY", question: "explain_function" }

  UNDO
    "Undo everything since I started this function"
    "Go back to working login"
    "Restore last checkpoint"
    → { type: "UNDO", target: "named_checkpoint", name: "working login" }

  Implementation:
    Send raw transcript + context to AI with structured output prompt.
    AI returns JSON. Parse and route to correct subsystem.

11.3 Layer 3 — Confidence Calibrator
---------------------------------------
Post-processes every AI suggestion before speaking to developer:

  High Confidence (>90%):
    AI states suggestion directly.
    "There's a null reference on line 14. I'm 91% confident.
     Here's my fix: [reads change]. Should I apply this?"

  Medium Confidence (60-90%):
    AI flags uncertainty but still suggests.
    "I see something on line 14 — I'm about 75% sure it's a null reference.
     Want me to walk you through it so you can decide?"

  Low Confidence (<60%):
    AI declines to suggest, proposes investigation.
    "Something looks unusual here but I'm only 45% confident.
     I'd rather we investigate together than guess. Want to explore it?"

  Implementation:
    Add to system prompt: "Always end your response with CONFIDENCE:XX
    where XX is your certainty percentage 0-100."
    Parse this tag from response. Remove before speaking. Use score
    to select response template above.

11.4 Layer 4 — Spatial Mapper
--------------------------------
Translates Tree-sitter AST nodes to audio parameters:

  Process:
  1. Tree-sitter parses file into AST
  2. For each node: extract type, depth, line range
  3. Map node type → audio texture
  4. Map depth → pitch rate
  5. Map line number → stereo position
  6. On cursor move: fetch node at cursor, compute audio params, play

  Node Type → Audio Texture Map:
    function_declaration  →  click (entry) + pitch rise
    if_statement          →  subtle tone shift
    for_statement         →  subtle tone shift
    error (LSP)           →  error-buzz.mp3 loop
    comment               →  chime.mp3
    import_statement      →  ping sound
    class_declaration     →  warm pad tone

11.5 Layer 5 — Session Memory
--------------------------------
Maintains conversation context tied to code locations:

  Memory Object (per session):
  {
    "changes": [
      { "timestamp": 1234567890, "type": "edit", "description": "added null check",
        "file": "checkout.js", "line": 14, "checkpoint": "null-fix-checkout" }
    ],
    "discussed_functions": ["processPayment", "validateUser"],
    "resolved_errors": ["null reference line 14"],
    "active_errors": ["type mismatch line 28"],
    "checkpoints": [
      { "name": "null-fix-checkout", "git_hash": "abc123", "timestamp": 1234567890 }
    ],
    "developer_preferences": {
      "explanation_depth": "detailed",
      "confirmation_style": "always_read_aloud"
    }
  }

  Inject full memory object into every AI context call.
  Persist to local JSON file during session.

12. TRUST PROTOCOL SPECIFICATION
================================================================================

The Trust Protocol is BlindCode's answer to the core fear:
"What if AI gives wrong code and the blind developer can't see it?"

4 Layers — All must pass before code changes:

Layer 1 — PROPOSE (Never Auto-Apply)
--------------------------------------
  - AI generates change
  - Change is stored in pending_change object
  - Nothing is written to file yet
  - AI speaks: "I'd like to [description]. Here's what I'll do: [reads change]"

Layer 2 — READ ALOUD (Full Transparency)
------------------------------------------
  - Before asking for confirmation, AI reads the EXACT change character by character
    if requested, or summarizes it clearly
  - Developer hears exactly what will happen
  - Commands available:
    "Read it again"    → re-reads the proposed change
    "Spell it out"     → reads character by character
    "Show context"     → reads 3 lines before and after the change

Layer 3 — CONFIDENCE SCORE (Calibrated Trust)
------------------------------------------------
  - AI states its certainty: "I'm 91% confident this is correct."
  - If <80%: "I'm not fully certain — this is my best guess at 73%."
  - Developer can factor this into their decision
  - Developer can ask: "What could go wrong?" → AI lists potential issues

Layer 4 — CHECKPOINT (Auto Safety Net)
-----------------------------------------
  - Before applying ANY change, auto-create a Git commit
  - Name: auto-generated from timestamp + change description
  - Developer can name it: "Call this checkpoint 'before big refactor'"
  - To restore: "Go back to 'before big refactor'"

Confirmation Commands:
  "Confirm" / "Yes" / "Apply it"     →  Apply change, announce completion
  "Reject" / "No" / "Don't do that"  →  Discard change, nothing changes
  "Modify it" / "Change the..."      →  AI revises proposal, starts over
  "Not now" / "Hold that"            →  Saves proposal for later

Trust Protocol Flow Diagram:
  Developer: "Fix it"
       |
       v
  AI generates fix
       |
       v
  [CREATE CHECKPOINT] ← silent, automatic
       |
       v
  AI speaks: "I'll [description]. Confidence: 91%. Here's the change: [reads it]"
       |
       v
  AI speaks: "Should I apply this?"
       |
       +---------> "Confirm"  →  Apply change → "Done. Checkpoint saved as [name]."
       |
       +---------> "Reject"   →  Discard → "Understood, nothing changed."
       |
       +---------> "Modify"   →  Revise → Start Trust Protocol again
       |
       +---------> silence    →  Wait indefinitely (never timeout and apply)

13.1 Voice Input (Speech to Text)
-----------------------------------
Tool: OpenAI Whisper (runs 100% locally, free, offline)

Setup:
  pip install openai-whisper
  Model: "base" model — good accuracy, fast on CPU (39M parameters)
  For better accuracy: "small" model (slower but worth it)

  import whisper
  model = whisper.load_model("base")
  result = model.transcribe("audio_chunk.wav")
  transcript = result["text"]

Wake Word Integration:
  - Porcupine detects "BlindCode" wake word
  - On detection: start 5-second recording window
  - Send recording to Whisper
  - Return transcript to Intent Parser

Push-to-Talk Fallback:
  - VS Code keybinding: hold Alt+B to record
  - Release to transcribe and process
  - No wake word needed

Noise Handling:
  - Whisper's local model handles moderate background noise
  - Add VAD (Voice Activity Detection) to trim silence
  - pip install webrtcvad for additional noise filtering

13.2 Voice Output (Text to Speech)
------------------------------------
Tool: Web Speech API (browser built-in, zero cost, zero latency)

Implementation (in VS Code Webview):
  function speak(text, priority = 'normal') {
    // Cancel current speech if high priority
    if (priority === 'high') {
      window.speechSynthesis.cancel();
    }
    const utterance = new SpeechSynthesisUtterance(text);
    utterance.rate = userPreferences.speechRate || 1.0;  // user-adjustable
    utterance.pitch = 1.0;
    utterance.volume = 1.0;
    // Select best available voice
    const voices = window.speechSynthesis.getVoices();
    utterance.voice = voices.find(v => v.lang === 'en-US') || voices[0];
    window.speechSynthesis.speak(utterance);
  }

Speech Commands (always available):
  "Repeat that"           →  Re-speaks last AI response
  "Slower"                →  speechRate -= 0.2
  "Faster"                →  speechRate += 0.2
  "Stop talking"          →  speechSynthesis.cancel()
  "Spell it out"          →  Reads last response character by character

15. DEBUGGING SYSTEM
================================================================================

15.1 Error Discovery
---------------------
Command: "Does anything look wrong?" / "Find bugs" / "What's broken?"

Process:
  1. Fetch all LSP diagnostics for current file
  2. Pass diagnostics + code snippet to AI
  3. AI ranks by severity, explains in plain English
  4. Reads most critical error first
  5. Asks: "Want me to walk you there?"

Response Template:
  "I found [N] issues. The most critical: [plain English description].
   I'm [X]% confident about this. It's at line [N], inside [function name].
   Want me to take you there?"

15.2 Error Navigation
----------------------
  "Yes" / "Take me there"  →  Cursor moves to error line
  On arrival: error-buzz.mp3 plays, AI reads the exact line
  "Explain this"           →  AI explains why this specific code is problematic

15.3 Plain English Stack Traces
---------------------------------
Command: "What happened?" / "Explain the crash" / "Replay the error"

AI converts raw stack trace to narrative:
  Raw: "TypeError: Cannot read property 'id' of undefined at processPayment (checkout.js:14)"
  Spoken: "The crash happened in processPayment. On line 14, it tried to read
           the 'id' field from an object called 'user', but user was undefined
           — it was never set before this function ran."

15.4 Variable State Narration
--------------------------------
Command: "What's in [variable]?" (during debug session)

Response: "user is currently undefined. It was last set on line 8 as the
          result of the getUserById call, which returned null."

15.5 Execution Replay
----------------------
Command: "Walk me through what happened step by step"

AI narrates execution path:
  "The function started. Line 8 called getUserById — that returned null.
   Line 11 called validateCart — that passed. Line 14 tried to access
   user.id, but user was null from line 8. That's where it crashed."

18. EDGE CASES & MITIGATIONS
================================================================================

EDGE CASE 1: AI generates wrong code
--------------------------------------
Risk: Blind developer cannot visually verify — wrong code gets committed
Mitigation:
  - Trust Protocol Layer 1: AI never applies without "confirm"
  - Trust Protocol Layer 2: AI reads every change aloud before applying
  - Trust Protocol Layer 3: Confidence score — developer knows certainty
  - Trust Protocol Layer 4: Auto checkpoint before every change
  - Developer can say "spell it out" for character-by-character reading
  - Developer can say "what could go wrong?" before confirming
  - All changes are one voice command away from reverting

Response to judges:
  "A sighted developer can also commit broken code. BlindCode has 4 safety
   nets that most sighted IDEs don't have."

EDGE CASE 2: Developer mishears the AI's spoken suggestion
------------------------------------------------------------
Risk: Developer hears wrong information, confirms wrong change
Mitigation:
  - "Repeat that" — re-speaks last response at same speed
  - "Slower" / "Faster" — adjusts speech rate permanently
  - "Spell it out" — character-by-character verification
  - TTS speed defaults to 0.85x (slightly slow) on first run
  - Every change confirmed → checkpoint created → instant rollback available

EDGE CASE 3: Noisy environment degrades voice input
-----------------------------------------------------
Risk: Whisper mishears commands, wrong actions triggered
Mitigation:
  - Push-to-talk mode: hold Alt+B, release to process (no mishearing)
  - Whisper base model filters moderate background noise
  - webrtcvad (Voice Activity Detection) trims silence and noise
  - All voice commands also available as keyboard shortcuts
  - Ambiguous commands ask for clarification: "Did you mean navigate or diagnose?"

EDGE CASE 4: AI API unavailable or rate limited
-------------------------------------------------
Risk: Free tier limits hit, AI stops responding mid-demo
Mitigation:
  - Primary: Claude API free tier
  - Fallback 1: Gemini 1.5 Flash free tier (automatic switch)
  - Fallback 2: Local lightweight model (Ollama + CodeLlama free)
  - For demo: pre-cache common responses, run against controlled test file
  - API calls are lightweight — free tier sufficient for 10-minute demo

EDGE CASE 5: "How is this different from Claude Voice / ChatGPT voice?"
------------------------------------------------------------------------
Claude Voice / ChatGPT Voice:
  - General chatbot with a microphone
  - No awareness of file structure, cursor position, or project
  - No spatial audio navigation
  - No connection to live errors or variable states
  - No trust protocol
  - One-shot Q&A, no session memory tied to code
  - Built for everyone

BlindCode:
  - Surgical IDE instrument
  - Knows exact cursor position in real time
  - Spatial audio gives code a 3D shape
  - Wired directly to VS Code LSP — sees live errors and execution
  - 4-layer trust protocol
  - Session memory tied to specific code locations
  - Every design decision made for someone who cannot see

"Asking how BlindCode differs from Claude Voice is like asking how
a surgeon's scalpel differs from a kitchen knife. Both are sharp.
Only one was built for the operating table."

EDGE CASE 6: "Isn't this a small market — blind developers are rare"
---------------------------------------------------------------------
Response:
  - 253 million visually impaired globally, 2.2 billion with some impairment
  - Zero IDEs were built for them — market is not small, tools have been small
  - Spatial audio navigation benefits sighted devs: commuting, multitasking,
    eye fatigue, vision disability accommodation at workplace
  - Historical precedent: voice interfaces designed for disability
    became Siri, Alexa, Google Assistant — mainstream consumer products
  - "We built for the extreme user. The mainstream will follow."

Response:

  1. Custom AST-to-audio spatial mapper (Tree-sitter + Howler.js)
  2. Intent parser converting speech to structured IDE commands

EDGE CASE 9: Large codebases — AI loses context

  - AI reads only ±10 lines of cursor context + error list

EDGE CASE 10: Latency — AI takes too long to respond
------------------------------------------------------
Target: < 3 seconds from voice command to spoken response
Mitigation:
